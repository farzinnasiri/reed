# Plan — Address Code Review Findings for Workout Feature

Scope: apply fixes for every issue surfaced in the code review of the uncommitted changes (workout surface, live sessions backend, catalog import, design system tweaks). Grouped by priority. Each task lists the files to touch, the change, and the verification step.

---

## Open questions to resolve before implementation

1. **`resumeAfterRest` vs `returnFromRest` semantics.** They are currently identical on the server. Confirm desired behavior:
   - Option A — Collapse into a single mutation `endRest`; UI distinguishes swipe-left vs swipe-right purely visually.
   - Option B — `resumeAfterRest` (swipe-right) clears `activeProcess` and marks that the user is starting the next set (no UI change there today, but leaves room for analytics / flags like "rest completed"). `returnFromRest` (swipe-left) clears `activeProcess` AND navigates user back to timeline (client-side), possibly keeping the rest record but marking it "abandoned" rather than deleted.
   - Recommended: **Option A** unless a real analytics/behavior split is planned.

2. **Session-switch rest preservation.** In `selectExercise`, should switching to a different exercise:
   - A) Preserve the rest card on the original exercise (only update `activeSessionExerciseId`), so rest continues in the background, OR
   - B) Clear `activeProcess` (current behavior)?
   - Recommended: **A** — matches the legacy background-rest model.

3. **`adjustBy` for rest timer:** confirm that `+/-15` should adjust `remainingSeconds` (keep current elapsed) and NOT reset to a fresh full duration. Recommended: yes, adjust only remaining.

4. **Bottom dock during active workout.** Currently the Coach/Workout segmented control disappears the moment a session starts. Confirm this is the intended "full-screen workout" behavior, or whether we should keep the dock visible and let the user swap tabs with the session still running in the background.

5. **`importCatalogBatch` invocation path.** Is the project comfortable moving the import script to use a Convex Node client (`ConvexHttpClient`) with a deploy key, or must we keep shelling out via `npx convex run`? AGENTS invariant #4 warns against ad-hoc `ConvexHttpClient` flows; the script is developer-run tooling, not app runtime, so it is arguably out-of-scope, but the team should sign off.

### P1.5 Fix `adjustBy` rest behavior
File: `convex/liveSessions.ts` — `updateRestProcess`, `adjustBy` branch

### P1.7 Normalize out-of-range values without re-render ping-pong
File: `components/home/workout-metric-picker.tsx`

Verification: no infinite loop warnings; picker renders fine when parent passes a value equal to `initialMetrics[key]`.
